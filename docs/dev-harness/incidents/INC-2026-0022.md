## INC-2026-0022 — 通知播放空 session 自递归 StackOverflowError

- id: INC-2026-0022
- area: player
- date: 2026-05-22
- status: active
- rule_ref: docs/dev-harness/player/rules.md#rule-notification-play-no-recursion
- guard:
    type: contract-test
    target: player/src/test/java/com/hank/musicfree/player/controller/PlayerControllerNotificationControlsTest.kt
- fix_ref: 见 `player/src/main/java/com/hank/musicfree/player/controller/PlayerController.kt` 中 `playFromNotification` 的 inline 注释；v1.2.3 引入，2026-05-22 修复

### 根因

v1.2.3 `PlaybackService.onPlayerCommandRequest` 新拦截「`COMMAND_PLAY_PAUSE` + `session.player.mediaItemCount == 0`」场景，回调 `PlayerController.playFromNotification`。原实现在 `mediaController.currentMediaItem != null` 时直接走 `play() → controller.play()`，而 `controller.play()` 经 Media3 IPC 又触发同一个 `onPlayerCommandRequest`（session player 仍为空），形成同步无限递归直到 `StackOverflowError`（线上堆栈见 2026-05-22 用户反馈）。

可触发的代表场景：MediaController 在控制端缓存了 `currentMediaItem`（例如 `setMediaItem` IPC 已在控制端写入本地缓存），但 session 侧 `player.mediaItemCount` 仍为 0（如 `clearMediaItems` / `stop` 之后、或异步分发未完成）。这种短暂不一致就让递归得以闭环。

### 复发条件

- 在 `PlaybackService.onPlayerCommandRequest` 的「空 player」分支中同步调用任何最终会回到 `mediaController.play()` / `controller.play()` 的方法。
- 在 `playFromNotification`（或同等的「通知触发的播放重启」入口）里基于 `mediaController.currentMediaItem` 判断后调 `play()` 而不是先 `setMediaItem` 再 `play`。

### 教训

- `onPlayerCommandRequest` 拦截的语义是「session 当前是空的，需要重新装载」。回调路径必须先 `setMediaItem` 让 session 拿到 item，再让 `play` 流过去，这样下一次 `onPlayerCommandRequest` 看到 `mediaItemCount > 0` 自动放行。
- 不能依赖 `mediaController.currentMediaItem`（控制端缓存）与 `session.player.mediaItemCount`（服务端 player 真实状态）在同一时刻一致。涉及「重新装载」语义时一律走 `activateCurrentQueueItem`。

### 备注

- contract-test 已加在 `PlayerControllerNotificationControlsTest.kt`：通过 `@VisibleForTesting setMediaControllerForTest` 注入一个 `currentMediaItem` 非 null 的 mock `MediaController`，断言走 `activateCurrentQueueItem`（resolver 被调用）且 `controller.play()` 不会被直接调用。
- 线上堆栈还原方式见 `docs/dev-harness/release-mapping.md`（v1.2.3 mapping 反查出 `wh2 = LoganMfLogger`、`gw.g = onPlayerCommandRequest → playFromNotification → play():215`，`fo3 = PlayerController`、`cr.a:358 = withConnectedController$lambda$0$0`，证实递归在 controller ↔ session IPC 之间）。
